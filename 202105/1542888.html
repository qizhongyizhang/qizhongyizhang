MySQL基础-库表数操作

写在前面：本文主要介绍MySQL的基础-数据库表、数据相关操作的基础知识

1. 数据库

创建数据库：create database [if not exists] db_name [数据库选项];

db_name：可以使用任意字符的组合，纯数字，特殊字符，mysql需要用反引号作为限定符标识，可以使用中文(不建议使用)；见名知意，推荐使用下划线命名法
数据库选择可以设定数据库字符集(character set utf8)和校对集(collate utf8_general_ci)等
创建成功，会在mysql的data目录下形成一个以数据库名为名称的目录，里面有一个db.opt文件，用于存储数据库的库选项信息

删除数据库：drop database [if exists] db_name; 删除一个数据库时，同时删除该数据库相关的目录及其目录内容

修改数据库：alter database db_name [修改指令(数据库信息的修改)];

例： alter database db_name character set gbk;

数据库改名：数据库不支持 rename 操作；
1. 新建数据库，插入源数据库数据；
2. 创建一个新数据库，将表都移动到新数据库

选择数据库：use db_name;

show databases [like ‘xxx’];  -- 列出所有数据库
show create database db_name  -- 查看创建数据库的DDL
show tables;  -- 列出默认数据库的所有表
show tables from db_name;  -- 列出指定数据库的所有表

2. 数据表

2.1 创建

创建数据表：创建一个表，会在数据目录创建对应的文件保存表信息；tb_name 命名规范与 db_name 相同，可使用逻辑前缀或后缀进行区分

create table [if not exists] tb_name(
列定义
) [表选项(引擎:engine|type,字符集:charset set,校对集:collate)]  [注释(comment)];

create table if not exists tb_user_info( 
user_id int NOT NULL AUTO_INCREMENT,  -- 设置自增长
user_name char(20) NOT NULL,
user_sex boolean DEFAULT 1,           -- 设置默认值，即默认值约束
user_address varchar(100),
user_Email char(30) unique            -- 设置唯一值约束
PRIMARY KEY(user_id),                 -- 申明主键
FULLTEXT(user_address)                -- 申明全文本索引
)ENGINE = InnoDB DEFAULT CHARSET=utf8mb4 ROW_FORMAT=DYNAMIC COMMENT = '客户信息表'

2.1.1 表选项

引擎类型：ENGINE ，也叫表类型，指的是数据表的存储机制，索引方案等配套相关功能；具体管理和处理数据的内部引擎，引擎在内部处理 SQL请求；MySQL 具有多种引擎，都隐藏在MySQL服务器内，为不同的任务选择正确的引擎以获得良好的功能和灵活性

选择表的存储引擎的标准

  • 是否需要事务支持
  • 是否支持外键
  • 并发量的要求
  • 错误恢复的处理
  • 是否支持某些具体功能

2.1.2 列定义

设置主键：PRIMARY KEY，数据唯一且不为空，支持多列组合主键，多列用逗号‘，’分隔；唯一标识一条记录，提高数据检索效率

create table tb_anme(
    col_name d_type(length) primary key,  -- 在定义表的时候直接在列定义时指明；
);

create table tb_anme(
    col_name d_type(length),
    primary key (col_name),  -- 在完成表定义时，在小括号内部指明主键列
);

例：
create table tb_anme(
    user_id char(9) primary key  -- 最后一个字段前不需要加逗号,
);

create table tb_anme(
    user_id char(9),
    user_name varchar(30),
    PRIMARY KEY(user_id, user_name)  -- 定义组合主键，（id， name） 组合构成主键
);

设置外键：FOREIGN KEY，关联其他表的主键，不能跨引擎使用，col_name 必须是其他表的主键；保证数据的完成性，提高数据检索效率

create table tb_anme(
    col_name foreign key references tb_name (col_name),
);

create table tb_anme(
    CONSTRAINT cons_name foreign key(col_name) references tb_name (col_name) [on delete set null],  -- 其余的列定义约束都有上述两种增加方式
);

**唯一约束(unique)**：该列的所有数据都唯一

col_name unique

user_id char(9) unique

指定默认值：DEFAULT，性别在未给出数据的情况下使用 1-男性；不允许使用函数作为默认值，只支持常量

col_name default value,

sex bool default 1,

**空值约束(NOT NULL)**：表示没有值，什么都没有，与任何数据不同；如果一个列不允许为空，但是在赋值时，没有为该字段赋值，则会报错；使用 is null or is not null来判断，不能直接比较=/>null

col_name [not null]

user_id char(9) not null, 表名 user_id 不允许为空，省略 not null 表明该列可以为空

设置自增长：AUTO_INCREMENT，每当增加一行时该列自动增长，每个表只允许一个AUTO_INCREMENT 列，且必须被索引；可通过 last_insert_id() 函数获得最后一个```

col_name AUTO_INCREMENT value

user_id char(5) AUTO_INCREMENT '00001', user_id 列从 00001 开始，每增加一行，数据自动加一

2.1.2.1 修改列定义

增加约束：对数据列添加相关的约束条件，满足数据使用需求；都支持在表定义的时候添加，也支持表定义后单独添加

alter table tb_name 
add constraint constraint_name constraint_type (column_name) 
[on delete/update cascade(级联操作)/restrict(不允许操作)/set null];

alter table test add constraint unique_constraint unique unique (id,name); 在test表的(id,name)组合增加唯一性约束，省略约束名
  • **主键约束(Primary Key)**：表中的一个或多个字段，唯一标识一条记录的字段集合，只能有一个主键；尽量不使用真实实体的属性，利用一个或多个与实体不相关的字段，作为唯一标识，主键不能为空，不能重复，看成一个唯一索引
alter table test add constraint P_Key primary Key (id);

删除主键:alter table tb_name drop primary key; 主键字段原则上在插入后，不应该修改，但是语法支持，要求修改后的值不能与已有值冲突
  • **外键约束(Foreign Key)**：如果A表的一个字段指向B表的主键，则此字段就为A表的外键。用于表示表之间的关系：存在外键的表，称之为从表(子表)，外键指向的表，称之为主表(父表)，一个表可以有外键，可以重复、可以为空；保持数据一致性，参照完整性，主要目的是控制存储在外键表(从表)中的数据；只有innodb存储引擎支持外键，其他引擎不支持
alter table tb_name_p add [foreign_key_name] foreign key (col_name) references tb_name_s(col_name);

ALTER TABLE audit ADD FOREIGN KEY (EMP_no) REFERENCES employees_test (ID); 在 audit 表的 EMP_no 字段上添加外键约束，引用 employees_test 表的ID字段，省略外键名

  • **唯一约束(unique)**：该列的所有数据都唯一

    alter table tb_name add constraint unique_name unique(col_name);

  • **自动增长约束(auto_increment)**：column_name auto_increment，整型，必须建立在索引之上，通常定义在主索引(主键)字段上；通常自动增长是从1开始递增，可以通过修改表属性，更改初始值

alter table tb_name auto_increment value; 可重复设置，新增数据从最大值开始增长，可以手动插入数据(可重复插入，可更新)

  • 级联操作约束：主表改变时，从表的反应

    1. cascade ：关联操作，从表进行相应操作：主表数据更新(主键值更新)，从表也更新(外键值更新)；主表记录删除，从表相关记录也删除。
2. set null：设为null，不在指向任何主表记录，主表数据被更新(主键值更新)，从表的外键被设置为null
3. restrict：拒绝，拒绝主表的相关操作

  • 检查约束(CHECK)

  • **默认值约束(DEFAULT)**：为列指定一个默认值，当该字段不存在值时启用；默认值必须是常数，不能是函数或表达式；如果该列可以为空，则默认值为null；blob，text数据列不能有默认值；时间戳类型，可以设置一个特殊的默认值 CURRENT_TIMESTAMP；在当列不存在或者传递的值为null时，使用当前的时间戳

  • **空值约束(NOT NULL)**：表示没有值，什么都没有。与任何数据不同。如果一个列不允许为空，但是在赋值时，没有为该字段赋值，则会报错。使用 is null or is not null来判断，不能直接比较=/>null

  • 注释：进行字段描述

    column_name comment ‘description’;
comment on column tb_name. column_name is ‘description’;

  • 其他：MySQL回味每一条记录加上三个属性：trx_id(事务ID)、roll_point(回滚指针，存储历史版本数据的地址)、row_id(行号)

    2.1.3 快速创建

    create table [if not exists] tb_name like old_tb_name;
create table [if not exists] tb_name select 结果集;

2.2 修改

删除表：表名不存在无法删除，可同时删除多个表，表名之间用逗号分割，没有确认，不能撤销

drop table [if exists] tb_name [CASCADE];

重命名：可同时重命名多个表，表名之间用逗号分割；剪切复制，可以跨库操作

rename table old_tb_name to new_tb_name[, old_tb_name to new_tb_name];

更新表：更新表定义，即表结构

增加：alter table tb_name add 列定义 after col_name|first（第一列）
删除：alter table tb_name drop col_name;
修改：alter table tb_name modify 列定义; -- 修改数据类型
重命名：alter table tb_name charge old_col_name new_col_name 列定义;

修改表选项：alter table tb_name 新表选项;

2.3 查看

查看所有表：show tables [like '_/%xxx'];
查看表的DDL：show create table tb_name;
查看表的描述性信息：show table status from db_name;
描述表：describe/desc tb_name;
查看列信息：show columns from tb_name;
查看索引信息：show index from tb_name;

2.4 视图

视图：是一个虚拟表，视图不保存数据，通过完成相应的SQL语句得到相应数据，视图的内容可以来自当前或其它数据库的一个或多个表，或者其它视图；视图常用来缩减业务逻辑，保存复杂逻辑，通常只对视图只进行简单逻辑操作

优点：

  • 视图保存查询的快照，可以简化查询操作，使复杂的查询易于理解和使用
  • 可以隐藏数据，提高数据的安全性，将用户的访问权限限制在列上，用户能以多种角度看待同一张数据表
  • 重用SQL语句，简化复杂的SQL操作
  • 重新格式化检索出的数据，满足使用数据的需要
  • 过滤掉不需要的数据
  • 为数据库提供一定程度的逻辑独立性

规则：

  • 视图名必须唯一，且不能与表重名；可以在查询时重命名列名作为视图的列名
  • 创建视图的数目没有限制
  • 视图可以嵌套，即可从视图中检索创建新视图
  • 视图不能索引，也不能有关联的触发器或默认值
  • 视图可以和表一起关联查询使用
  • 创建视图中拥有 where 子句，基于视图的查询也拥有 where 子句，二者将会自动组合过滤
  • 可以进行插入、更新、删除，最终还是操作基表，即时生效，如果 MySQL 不能正确地确定被更新的基数据，则不允许更新，包括插入和删除
  • 视图定义中有 分组（使用GROUP BY和HAVING）、联结、子查询、并、聚集函数（Min()、Count()、Sum()等）、DISTINCT、导出（计算）列等不能进行更新操作

2.4.1 操作

创建视图

create [OR REPLACE] [ALGORITHM = {UNDEFINED | MERGE | TEMPTABLE}] view view_name as (select result);

create view actor_name_view as select first_name first_name_v,last_name last_name_v from actor; 对 actor 表创建仅有first_name、last_name 两列，且视图字段名为 first_name_v、last_name_v 的actor_name_view 视图

删除视图

Drop view [if exists] view_name;

修改视图：通过 algorithm 指定视图执行算法：

  • Merge 合并，每当执行时，将视图的sql语句于外部操作语句混合在一起，最终执行
  • Temptable：临时表：每当执行时，将创建视图的sql语句生成一个临时表，之后对临时表进行操作，undefine进行创建
Alter view view_name as new_select_result;

查看结构

SHOW CREATE VIEW view_name;

3. 数据

3.1 导入

  1. 在登录mysql的情况下：source 文件路径/文件名
  2. 在不登录的情况下：mysql -u 用户名 -p 密码 库名 < 备份文件
  3. 导入：
Load data infile ‘f_name’   -- 数据文件
[REPLACE | IGNORE]     -- 主键冲突： Ignore(忽略) | REPLACE(替换)
INTO TABLE tb_name     -- 目标表
[FIELDS [TERMINATED BY 'string'] [[OPTIONALLY] ENCLOSED BY 'char'] [ESCAPED BY 'char']] -- 字段切割：fields terminated by '\t' enclosed by '' escaped by '\\'
[LINES [STARTING BY 'string'] [TERMINATED BY 'string']]   -- 记录行切割：lines terminated by '\n' starting by ''
[IGNORE N LINES]   -- 跳过n行数据

按照指定数据格式导入(默认)：字段使用逗号分割，而使用引号包裹

3.2 导出/备份

命令行实用程序：mysqldump

  • 导出一张表：mysqldump -u用户名 -p密码 库名 表名 > 文件保存路径/文件名
  • 导出多张表：mysqldump -u用户名 -p密码 库名 表名1 表名2 表名3 > 文件保存路径/文件名
  • 导出所有表：mysqldump -u用户名 -p密码 库名 > 文件保存路径/文件名
  • 导出一个库：mysqldump -u用户名 -p密码 -B 库名 > 文件保存路径/文件名
  • 使用-w指明备份条件

MySQL语句：BACKUP TABLE 或 SELECT INTO OUTFILE，导出查询结果到文件；
SELECT INTO OUTFILE：自动创建文件，但不能重写已有文件，默认采用换行符区分记录，制表符区分字段；支持配置字段分隔符、包裹符和转义符；配置行(记录)结束符和起始符，常使用逗号分割字段，引号包裹；导出的数据文件可以使用 LOAD DATA INFILE 语句载入到某个数据表内；支持 INTO dumpfile，作用与 outfile 一致，不做任何换行和转义处理导出结果到文件；常用来保存二进制数据；数据可以用 RESTORE TABLE 复原

select * INTO outfile ‘文件地址/文件名’   -- 目标文件
fields(字段) terminated by ‘，’ starting by ‘aaa’ enclosed by  ‘bbb’ escaped by '\\'    -- 字段切割
lines(记录) terminated by ‘\n’   -- 记录行切割
from tb_name;   -- 源数据表

每一行为一条数据，每个字段之间使用水平制表符分隔

3.3 插入

INSERT [LOW_PRIORITY] INTO tb_name (col_name_list) 
values (data_list) 
[on duplicat key update col_name = data[,col_name= data]]  (没有set); 

LOW_PRIORITY：降低 INSERT 语句的优先级
Into：可以省略(不建议)
  • INSERT INTO tb_name values (data_list);
    不明确指定列名，语句高度依赖于表中列的定义次序，并不安全，应该尽量避免使用
  • INSERT INTO tb_name (col_name_list) values (data_list);
    将位置相同的数据和字段一一对应，必须对每个列出的列给出一个值；即使表的结构改变，语句仍能正确工作
  • INSERT INTO tb_name (col_name_list) values (), (), ();
    通过记录构造符，一次性插入多个值；批量插入时推荐，效率优于一条一条单独插入
  • INSERT INTO tb_name SELECT …;
    将查询结果作为数据插入，要求结果的列名、数据类型与目标表一致，蠕虫复制
  • INSERT INTO tb_name set field=value,…;
    使用set语法
  • INSERT INTO tb_name values (field_value, 10+10, now());
    使用表达式
  • INSERT INTO tb_name values (field_value, default);
    使用特殊值 default
  • INSERT IGNORE INTO tb_name (col_name_list) values (data_list);
    出现主键冲突时忽略冲突记录不插入，插入其他不冲突的记录
  • INSERT INTO tb_name (col_name_list) values (data_list) on duplicate key update col_name = data, …;
    指定在插入的值出现主键(或唯一索引)冲突时，更新其他非主键列的信息

3.4 删除

delete from tb_name [WHERE where_condition] [ORDER BY ...] [LIMIT N];

  • 支持多表删除：Delete from tb_name_1,tb_name_2 using 表连接条件

  • TRUNCATE [TABLE] tb_name; 清空数据，删除重建表

  • 区别：

    1. truncate是删除表再创建空表，delete是逐条删除
    2. truncate会重置 auto_increment 的值，delete不会
    3. truncate不返回删除条数，delete会返回

3.5 修改

update [LOW_PRIORITY] [IGNORE] tb_name set col_name = data [WHERE where_condition] [ORDER BY ...] [LIMIT N];

IGNORE：忽略插入过程中的错误，继续更新；正常出现错误就会把错误发生前更新的所有行恢复到原来的值
支持更新多列，用逗号分隔

多表更新：Update [LOW_PRIORITY]  [IGNORE] t1 join t2 on t1.x=t2.x where cond;多表更新时ORDER BY和LIMIT不能被使用；
  • 替换： REPLACE INTO tb_name values (data_list); 存在更新，不存在插入。但如果表中的一个旧记录与一个用于PRIMARY KEY或一个UNIQUE索引的新记录具有相同的值，则在新记录被插入之前，旧记录被删除(替换原有记录)
  • 数据更改操作流程：数据产生变更时，会先在内存更改数据，当在特定时期再写入磁盘(频繁的I/O操作会降低性能)

脏页：指的是存储引擎内存中的数据与磁盘中的数据不一致，进行数据更改操作(增删改)时，会先写预写日志(WAL)，再更改内存中的数据，最后在合适时机进行脏页落盘

  1. 将数据加载到Buffer Pool中
  2. 生成修改操作的redo log日志，写入redo log Buffer 缓冲(为降低I/O操作，提高效率)
    • 写日志的效率优于写数据的日志，所以先写日志，再写数据
    • redo log Buffer缓冲，采用 force log at commit 保证 redo log Buffer 在MySQL服务器宕机时将数据写入磁盘，force log at commit要求事务提交，立即刷新redo log Buffer，可通过 innodb_flush_log_at_trx_commit 参数进行设置，有效值：0-立即更新磁盘数据，1-每秒提交一次，默认，2
  3. 当redo log Buffer 缓冲在指定的时间将缓冲区的 redo log 写入磁盘上的 id_logfile文件中
  4. 当磁盘上的 id_logfile文件写满时，会触发checkpoint机制，将Buffer Pool里的数据写入磁盘，清空id_logfile文件，循环使用
  5. 当Buffer Pool缓冲满时，也会触发checkpoint机制(控制脏页何时落盘)，将Buffer Pool里的数据写入磁盘
    • 保证脏页落盘，使用Double Write(双写日志) ：MySQL数据页的大小是16k，而操作系统的磁盘块大小是4k，需要四次写入，双写日志保证数据页数据更改失败时能够保证数据一致
    • Double Write由两部分组成：Double Write Buffer(大小2M)和物理磁盘上共享表空间连续的128个页，大小也为2M。
    • 对缓冲区的脏页进行刷新时，先通过memcpy()函数将脏页先复制到内存中的double write buffer区域，分两次(每次1M)顺序写入共享空间的磁盘文件上，然后调用fsync()函数，同步磁盘，避免操作系统缓冲带来的问题。完成double write 页的写入后，再将double write buffer区域的页写入各个表空间文件
