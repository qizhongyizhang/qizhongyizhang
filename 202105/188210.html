写在前面：本文主要记录 MySQL进阶-查询优化策略 的相关基础知识

1. 查询效率

1.1 影响因素

执行时间长，等待时间长

  • 服务器硬件设备性能：发生在数据装入内存或从磁盘上读取数据；可通过 top、free、iostat、vmstat 查看系统性能
  • 磁盘IO数：磁头移动到指定磁道的时间花费，发生在装入数据远大于内存容量
  • 参数配置不合理
  • 数据量的多少
  • SQL效率慢
  • 索引失效
  • 太多 join 关联查询

1.2 外部表现

  • 大量请求阻塞
  • SQL操作变慢
  • 存储出现问题

2. 性能分析

2.1 MySQL Query Optimizer

MySQL Query Optimizer：专职负责 select语句 的优化器模块，通过分析系统中收集到的统计信息，为客户端请求的 Query 提供自认为最优的执行计划

  1. 将从命令解析器解析过滤的 select语句 的常量表达式换算成常量，处理常量表达式的预算（Optimizer仅接收命令解析器模块过滤的 select语句）；
  2. 对 select语句 中的查询条件进行简化和转换，如去掉无用或显而易见的条件、结构调整等；
  3. 分析 select语句 中的 Hint信息，判断显示 Hint信息 能否完全确定 select语句 的执行计划；如果不能或没有 Hint信息，则会读取所涉及对象的统计信息，根据 select语句 进行相应的计算分析，然后得出最后的执行计划

2.2 执行计划

执行计划：explain SQL_list，在查询语句之前加上 explain 关键字可以模拟优化器执行SQL语句，显示出执行信息，包括 表的读取顺序、数据读取操作的操作类型、可以使用和实际使用的索引、表之间的引用、被优化器查询记录数；用于分析SQL语句的性能，只是在正常的SQL语句之前加上 explain 关键字，仅用于数据的增删改查类型的SQL，不涉及数据库、数据表和数据列的相关操作

指标名称 指标含义 指标取值及含义
id select子句或表的读取操作顺序 1、2、3…
id值相同，从上往下顺序执行；id值不同，id值大优先查询(先查内层-子查询，再查外层)

SUBQUERY：包含子查询SQL中的子查询（非最外层）
DERIVED：from列表中包含的子查询被标记为 derived（衍生查询），MySQL递归执行并将结果存放在临时表中
UNION：union 之后出现的 select
UNION RESULT：对 union合并结果 进行的 select
table 执行的表名 tb_name
partitions 使用到的分区 partitions_list

const：通过索引一次性找到；一般用于比较 primary key 、unique索引，只匹配一条数据，可以很快将主键至于 where列表中，MySQL可以将查询转换成一个常量
eq_ref：通过唯一性索引扫描，对于每一个索引值，表中只有一条记录匹配；常用于主键或唯一索引扫描
ref：通过非唯一性索引扫描，对于每一个索引值，表中有多条记录匹配
range：只检索给定范围的行，使用索引选择行，key显示使用的索引列；一般出现在 where语句 中存在 between、<、>、in 等
index：全索引扫描，只遍历索引树，从索引中读取数据
ALL：全表查询，从磁盘读取数据
查询效率从上到下，依次递减；一般操作中至少需要达到 range级别

key_len 索引中使用的字节数 正整数，通过该列计算查询中使用的索引长度，保证精度的前提下，长度越短越好
ref 显示索引的哪一列被使用 表名用于查找索引的列
rows 查询的行数 正整数，根据表统计信息和索引选用情况，估算出找到指定记录所需读取的行数，越小越好
filtered 返回的结果行占需要读到的行的百分比 0、1、2…100
extra 包含不适合在其他列显示但是很重要的额外信息 using index：使用到了索引
using where：使用了where条件
using join buffer：使用了连接缓存
impossible where：where子句 的值总是 false，不能用来获取元组
select tables optimized away：在没有 group by子句的情况时，基于索引优化 min/max操作或者count(*)操作
distinct：在找到第一个匹配的元组后即停止
using filrsort（九死一生）：无法利用索引完成的排序操作（称为”文件排序”），数据使用一个外部索引进行排序
using temporary（十死无生）：对查询结果排序时使用临时表保存中间结果，常见于 group by 和 order by

2.3 Show Profile

Show Profile：MySQL提供可以用来分析当前绘画中SQL语句执行的资源消耗情况，可以用于SQL调优的测量，官网，默认关闭，并保存最近15次测运行结果

show variables like 'profiling';  -- 查看是否具备功能

set profiling = on;  -- 开启 profiling

show profiles;  -- 查看SQL运行列表，包括SQL语句的 query_id、持续时间和SQL语句内容

查看系统消耗：

  • all：显示所有的系统消耗信息
  • block io：显示I/O相关的开销
  • context switches：显示上下文切换的开销
  • cpu：显示cpu的开销
  • ipc：显示发送和接收的开销信息
  • memory：显示内存的开销信息
  • page faults：显示页面错误的开销信息
  • source：显示和 source_function、source_file、source_line 相关的开销信息
  • swaps：显示次数交换相关的开销信息
show profile cpu, block io for query query_id;  -- 查看系统响应，存在以下内容，SQL效率不高
-- converting heap to MyISAM：查询结果太大
-- create tmp table：创建临时表
-- copying to tmp table on disk：江北村的临时表复制到磁盘
-- locked：表锁定

2.4 问题定位

系统状态

show status ： 显示状态信息（扩展 show status like 'xxx', show global status like '%slow%'）
show variables ： 显示系统变量（扩展 show variables like 'xxx'）
show innodb status ： 显示InnoDB存储引擎的状态
show processlist ： 查看当前 SQL 执行，包括执行状态、是否锁表等
mysqladmin variables -u username -p password ： 显示系统变量
mysqladmin extended-status -u username -p password ： 显示状态信息

开启慢SQL记录：在配置文件 my.cnf 或 my.ini 中在[mysqld]一行下面加入配置参数

slow_query_log = on     # 开启慢查询日志，slow_query_log = 1
long_query_time = 2     # 规定语句执行查过两秒就会记录，最好在5秒以内
log-slow-queries=/data/mysqldata/slow-query.log       # 规定慢查询日志文件存放位置，slow_query_log_file = /path/to/log

连接数：connections

show variables like 'max_connections'; 
show global status like ‘Max_used_connections’;
Max_used_connections / max_connections * 100% ≈ 85%

索引丢失率：key_buffer_size，Key_read_requests，Key_reads，key_cache_miss_rate

show variables like ‘key_buffer_size’;   # 查看索引缓存大小
show global status like 'Key_read_requests';     # 查看索引请求数
show global status like 'Key_reads';     # 查看索引未命中数
key_cache_miss_rate = Key_reads / Key_read_requests * 100%  # 索引丢失率，在0.1%以下较好(每1000个请求有一个直接读硬盘)，如果key_cache_miss_rate在0.01%以下的话，key_buffer_size分配的过多，可以适当减少。

临时表创建：Created_tmp_disk_tables ，Created_tmp_tables ，max_heap_table_size 

show global status like 'Created_tmp_disk_tables'; # 查看在磁盘上创建临时表数量
show global status like 'Created_tmp_tables';      # 查看创建临时表数量
show global status like 'Created_tmp_files';       # 查看创建临时文件数
Created_tmp_disk_tables / Created_tmp_tables * 100% <= 25% 

show variables where Variable_name in ('tmp_table_size', 'max_heap_table_size'); 
max_heap_table_size  # 内存临时表大小，规定值以下的临时表才能全部放内存
tmp_table_size       # 临时表大小，超过 max_heap_table_size 但在规定值以下创建磁盘临时表

打开表情况：Open_tables，Opened_tables，table_cache

show global status like 'open%tables%'; 
show variables like 'table_cache'; 
如果Opened_tables数量过大，说明配置中table_cache(5.1.3之后叫做table_open_cache)值可能太小

Open_tables / Opened_tables * 100% >= 85%
Open_tables / table_cache * 100% <= 95%

进程使用情况：Threads_created ，thread_cache_size，当客户端断开之后，服务器处理此客户的线程将会缓存起来以响应下一个客户而不是销毁

show global status like 'Threads_created';
show variables like 'thread_cache_size'; 

如果 Threads_created值过大，表明MySQL服务器一直在创建线程，比较耗资源，可以适当增加配置文件中thread_cache_size值

查询缓存：qcache、query_cache

show global status like 'qcache%'; 
show variables like 'query_cache%'; 
查询缓存碎片率 = Qcache_free_blocks / Qcache_total_blocks * 100%；查询缓存碎片率超过20%，可以用FLUSH QUERY CACHE整理缓存碎片，如果查询都是小数据量的话可以减小query_cache_min_res_unit
查询缓存利用率 = (query_cache_size - Qcache_free_memory) / query_cache_size * 100%；查询缓存利用率在25%以下的话说明query_cache_size设置的过大，可适当减小；查询缓存利用率在80%以上且Qcache_lowmem_prunes > 50 说明 query_cache_size 可能有点小，或者是碎片太多
查询缓存命中率 = (Qcache_hits - Qcache_inserts) / Qcache_hits * 100%；命中率较差，可能写操作比较频繁，而且可能有些碎片

查询排序：Sort_buffer_size ，Sort_merge_passes 

show global status like 'Sort_merge_passes'; 
show variables like 'Sort_buffer_size'; 
如果 Sort_buffer_size 的大小不够把所有的记录都读到内存中，MySQL 就会把每次在内存中排序的结果存到临时文件中，等 MySQL 找到所有记录之后，再把临时文件中的记录做一次排序；这次排序就会增加 Sort_merge_passes；实际上，MySQL 会用另一个临时文件来存再次排序的结果，通常 Sort_merge_passes 增加的数值是建临时文件数的两倍

文件打开数：open_files、open_files_limit

show global status like 'open_files';
show variables like 'open_files_limit';
比较合适的设置：Open_files / open_files_limit * 100% <= 75%

表锁情况：Table_locks_immediate ，Table_locks_waited 

show global status like 'table_locks%';
Table_locks_immediate：表示立即释放表锁数
Table_locks_waited：表示需要等待的表锁数
如果Table_locks_immediate / Table_locks_waited > 5000，最好采用InnoDB引擎

表扫描情况：Handler_read_rnd_next，Com_select、read_buffer_size

show global status like 'Handler_read_rnd_next';  # 显示查询直接操作的数据文件数量
show global status like 'com_select'; # 调出服务器完成的查询请求次数

表扫描率 = Handler_read_rnd_next / Com_select：如果表扫描率超过4000，说明进行了太多表扫描，很有可能索引没有建好，可以适当增加 read_buffer_size 值，但最好不要超过8MB

3. 优化策略

3.1 SQL层面-SQL优化

尽量避免全表扫描和笛卡尔积；优先优化内层循环；不要用 select *，只查询所需要的字段

join优化：

  • 使用 join 代替 子查询；尽可能减少 join语句 的循环总数，数据量较小的表为主表，关联数据量较大的副表，可在附表上建立索引，小表驱动大表
  • 无法保证副表 join字段被索引且内存资源充足的前提下，可以增加 join buffer 的大小

where优化：

  • where 子句中，表连接写在最前面，可以过滤掉大量数据的条件写在最右边，having写在最后面，条件匹配从右向左依次进行
  • 尽量避免在 where 子句中进行 null判断 和表达式操作，会导致放弃使用索引进行全表扫描
  • 避免在索引列上进行计算和 使用null判断(is [not] null)，会导致放弃使用索引进行全表扫描
  • 通配符（like）和正则表达式匹配会尝试匹配表中所有行，应尽量避免使用
  • 使用 exists 替代 in；[not] in -> [not] exists

group by优化：分组之前先进行排序

  • 使用索引列进行分组，且遵循最佳左前缀法则
  • 增大 sort_buffer_size参数 和 max_length_for_sort_data参数的设置
  • 优先在 where条件下进行数据限定，再通过 having进行数据限定

order by优化：尽量使用 index 索引排序，避免使用 Using filesort 文件排序（会创建临时表存储数据，再进行排序），即尽量使用创建索引的字段进行排序（B+Tree索引创建完成后默认已经排好序，不需要再通过数据表进行排序），当排序字段与索引字段不匹配时会采用 filesort排序方式

  • 查询字段避免使用 select *，严格限定查询的字段

  • 使用索引列进行排序，且遵循最佳左前缀法则；where使用索引的最左前缀定义为常量，order by会使用索引，例：index a_b_c, where a = const order by b, c

  • 排序字段的顺序与创建复合索引的顺序一致，且所有的排序字段排序方式一致，即全为增或全为减

  • 增大 sort_buffer_size参数的设置：双路排序算法、单路排序算法提高参数都会增加效率

  • 增大 max_length_for_sort_data参数的设置：select查询字段大小的总和小于 max_length_for_sort_data，且排序字段不是 text、blob等类型时，会采用单路算法，否则还是会用多路算法；提高参数，会增加用单路排序算法的概率；如果 max_length_for_sort_data设得太高，会增加数据量超出 sort_buffer_size的概率

    Using filesort 文件排序实现算法：

    • 双路排序：MySQL4.1之前使用双路排序，两次扫描磁盘，最终得到数据：从表中读取行指针和 order by列并进行排序，然后扫描已经排好序的列表，按照列表的数据重新从列表中读取数据输出；从磁盘读取排序字段，在 buffer中进行排序，再从磁盘获取其他字段
    • 单路排序：从磁盘读取需要查询的所有列，按照 order by列 在 buffer中进行排序，然后扫描排序后的列表进行输出；避免了第二次读取数据，并且把随机IO变成顺序IO；但把每一行都保存在内存中会消耗更多的空间，可能会因为 sort_buffer的容量小于读取的数据量，导致创建临时表，多路合并，造成多次I/O，会导致性能下降，甚至低于 双路排序算法
select distinct t1.a from t1,t2 where t1.a = t2.a;
select t1.a from t1,t2 where t1.a = t2.a group by t1.a order by t1.a;
  • distinct 替换为 group by，并结合 order by 使用
  • 用 union 代替手动创建的临时表

3.2 表结构层面

3.2.1 表结构层面-索引

索引：是帮助数据库高效获取数据的数据结构，是数据库优化中最常用也是最重要的手段之一，数据量越大，索引查询效率提高越明显，更多内容可以查看MySQL进阶-索引

优点：

  • 提高数据检索的效率，降低数据库的 I/O成本
  • 提高数据排序的效率，降低CPU的消耗

缺点：

  • 索引也是一张表，里面存储主键和索引字段，并指向实体记录；会占用额外的存储空间
  • 对表进行增删改时，索引也会动态维护数据更新后的索引信息，包括主键、索引字段和指向信息等，降低了数据的维护速度
  • 需要根据实际业务情况，不断更新重建最合适的索引
  • 在最频繁使用以缩小查询范围、频繁使用需要排序的字段上建立索引；不宜在很少在查询中涉及或者重复值比较多的列或者数据类型是文本的字段上建立索引

3.2.2 表结构层面-数据类型

  • 选取最实用的字段属性，如省份、性别最好用 enum 枚举类型

  • 尽可能减少字段宽度

  • 尽量把字段设置成 not null

    字段类型优先级：int -> date、time - > enum、char -> varchar -> blob、text；优先考虑数字类型，其次是日期或者二进制类型，最后是字符串，同级别数据类型优先选择占用空间小的数据类型

3.2.3 表结构层面-表冗余

将需要关联查询的数据冗余保存到需要关联的表内

3.2.4 表结构层面-创建分区表

  • 创建条件：数据量巨大

    3.2.5 表结构层面-锁机制

    适用外键，优化锁定表；锁定表，优化事务处理
  • 读写分离
  • 分段加锁
  • 减少锁持有时间
  • 多个线程尽量以相同顺序获取资源

    不能将锁的粒度过于细化，否则会出现线程加锁和释放的次数过多，从而降低效率

3.3 数据库配置

3.3.1 数据库配置-集群

3.3.2 数据库配置-主从复制

3.3.2.1 介绍

主从复制基本原理：slave 从 master 读取 binlog 进行数据同步

主从复制过程：

  1. Master 将数据改变记录到二进制日志：Binary Log；记录过程叫做二进制日志事件：Binary Log Events；
  2. Slave 将 Master 的 Binary Log Events 拷贝到它的中继日志：Replay Log；
  3. Slave 重做中继日志中的事件，将改变应用到自己的数据库中；MySQL复制是异步且串行化的

主从复制基本原则：

  • 每个 Master 可以有多个 Salve，每个 Slave 只有一个 Master
  • 每个 Slave 只能有一个唯一的服务器ID

3.3.2.2 操作

主从复制前置要求：

  1. MySQL版本一致，且后台以服务运行
  2. 多台MySQL服务器必须在同一网段，保证网络通畅

主从复制配置：在配置文件中完成配置，配置完成后需要重启MySQL服务器

  • Master
[mysqld]
server-id=1              # 主服务器的唯一ID，必须
log-bin=/path/mysql-bin  # 开启二进制日志，默认会生成 mysql-bin.000001, mysql-bin.000002...序列文件，必须
log-err=/path/mysql-err  # 启用错误日志，默认会生成 mysql-err.err 错误日志文件
basedir="/path/"         # 配置MySQL的根目录，basedir="C:/Program Files/MySQL/MySQL Server 8.0/"
tmpdir="/path/"          # 配置MySQL的临时目录，可与根目录相同
datadir="/path/"         # 配置数据目录，默认为 /根目录/data/，即C:/Program Files/MySQL/MySQL Server 8.0/data/
read-only=0              # 表示主机可读可写
binlog-ignore-db=db_name # 设置不进行复制的数据库
binlog-do-db=db_name     # 设置需要进行复制的数据库
  • Slave
[mysqld]
server-id=2              # 从服务器的唯一ID，必须

  • 主机创建账户并授权
GRANT REPLICATION SLAVE ON *.* TO 'zhangsan'@'slave_ip' IDENTIFIED BY '123456';

FLUSH PRIVILEGES;    -- 刷新权限
show master status;  -- 查看最新的 File 和 Position并记录
  • 从机配置需要复制的主机
CHANGE MASTER TO MASTER_HOST='master_ip',
MASTER_USER='zhangsan',
MASTER_PASSWORD='123456',
MASTER_LOG_FILE='mysql-bin.File_num',
MASTER_LOG_POS=Position_num;

start slave;   -- 开启 Slave从机的复制；stop slave，关闭从机的复制

show slave status\G  -- Slave_IO_Running 和 Slave_SQL_Running 同时为Yes，主从复制配置成功，否则失败

3.3.5 数据库配置-分库分表

使用场景：单表数据超过500万或者表容量超过2G，推荐进行分库分表，如果预计三年后的数据量达不到级别，不要在创建时就分库分表；有 垂直切分（按照业务切分，表结构可能不同）和 水平切分（表结构相同，按照主键ID、时间等进行分表）两种

3.3.5.1 名词介绍

逻辑表：逻辑上存储数据的表
真实表：实际存储数据的表
广播表：所有的库里面存储相同的数据
绑定表
分片键：进行分库分表的数据列
分片算法：将数据表以分片键进行分库分表的规则

随机：random
范围：range
hash（取模）：一致性hash
