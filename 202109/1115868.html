写在前面：本文主要对 MySQL的基础通识知识 进行简单介绍

1. 简单介绍

1.1 逻辑架构

连接层：

Connectors，提供与客户端的连接服务，主要包括连接处理、授权认证、安全方案等，引入线程池的概念，为通过安全认证的连接提供线程

服务层：提供SQL接收接口（针对 select 语句会优先进行缓存查询），随后完成SQL的解析（形成解析数）和优化（索引、连接顺序等，形成优化后的解析数）及部分内置函数的执行

  • Connection Management：连接池，进行连接管理

  • SQL Interface：SQL接口

  • Parser：解析器

  • Optimizer：优化器

  • Caches & Buffers：缓存和缓冲

  • Management Service & Utilities：

存储引擎层：真正实现 MySQL 的数据存储和提取，较常用的是 MyISAM（5.5及之前默认） 和 InnoDB（5.5 之后默认，可进行配置）

Pluggable Storage Engines(可拔插存储引擎)，插件式的存储引擎架构将查询处理、其他系统任务和数据存储提取相分离，可根据合适的业务场景选择哈市的存储引擎

show engines;                            -- 查看现在提供的存储引擎列表，Support中为default为当前默认的存储引擎
show variables like "%storage_engine%";  -- 查看当前默认提供的存储引擎

数据存储层：将数据存储在硬件设备上，并完成与存储引擎的交互

File System(文件系统)

磁盘文件

查看文件存放位置：show variables like ‘datadir’; 最多可以创建20亿个表，每个表最多1024个列，记录数没有明显限制（受到存储空间的限制）

1.2 表格类型

1.2.1 常见五种类型

  • MyISAM

  • InnoDB

  • ISAM：简称为索引顺序访问方法，由 IBM 开发用于在磁带等辅助存储系统上存储和检索数据

  • Merge

  • Heap：内存表；存储于内存中，用于临时高速存储，存储引擎是memory；表结构存储在磁盘中，数据存储在内存中，服务器重启后，仅保留 heap 表结构；存储引擎 memory 比 myisam 快

  • ISAM：简称为索引顺序访问方法，由 IBM 开发用于在磁带等辅助存储系统上存储和检索数据

    表的大小通过 Mysql 配置变量 max_heap_table_size 来控制；不允许存储 BLOB 或 TEXT 数据类型；只能使用比较运算符 <=> ；4.1版本之前不支持 AUTO_INCREMENT ；索引不可为 NULL

CREATE TABLE `tb_name` (
`col_name` datatype(long) 列属性选项
) TYPE = HEAP(5.2以前) / engine = memory(5.2以后)

表结构定义与普通表一致，仅在后面申明 TYPE = HEAP / engine=memory；添加主键、索引方式也与普通表一致

1.2.2 其他类型

  • Federated：访问在远程数据库的表中的数据；在本地构建一个federated表来连接远程数据表，本地表的数据可以直接跟远程数据表同步；引擎里面是不真实存放数据的，所需要的数据都是连接到其他MySQL服务器上获取
    注意事项：
    1. 对本地虚拟表的结构修改，并不会修改远程表的结构
    2. truncate 命令，会清除远程表数据
    3. drop命令只会删除虚拟表，并不会删除远程表
    4. 不支持 alter table 命令
    5. limit 操作运行速度较慢，读取所有满足条件的记录，再进行 limit 处理
install plugin federated soname 'ha_federated.so';  -- 安装federated引擎
在my.cnf中加入一行federated；保存重启  -- 启动federated引擎

CREATE TABLE tb_name (
`col_name` datatype(long) 列属性选项
) ENGINE = FEDERATED CONNECTION = 'mysql://username:password@ip:port/db_name/tb_name'
表名可以不一致
  • 临时表：默认存储引擎是 myisam，表结构、数据都存储在内存中，服务器重启后，都将会丢失
    CREATE TEMPORARY TABLE `tb_name` (
`col_name` datatype(long) 列属性选项
)
  • dual：伪表，不真实存在

单表最大的存储空间是 65535TB，单列最大的存储空间4GB（仅 LONGBLOB、LONGTEXT 支持）

1.3 存储引擎

存储引擎：文件系统的实现，数据、索引和其他对象的存储方式

1.3.1 MyISAM

不支持事务，但每次查询都是原子的，不支持外键
支持表级锁（对整表加锁）
按照记录插入顺序保存，存储表的总行数
一个MyISAM有三个文件：表结构文件（frm）、索引文件（myi）和数据文件（myd）；数据可被压缩，存储空间较小；跨平台转移时较方便，备份恢复时可单独对某个表进行操作
采用非聚簇索引，索引文件的数据域存储指向数据文件的指针，辅索引与主索引基本一致，但辅索引不用保证一致性

1.3.2 InnoDB

存储结构逻辑上可分为：表空间tablespace，段segment，区extent，页page，行row(一条记录)

表空间：

  • 系统表空间：也叫共享表空间，关闭独占表空间后：innodb_file_per_table = ，所有的数据都会存储到ibdata1文件里面
  • 用户表空间：也叫独占表空间，开启独占表空间后：innodb_file_per_table = 1，每张表的数据都会存储到一个独立的ibd文件里面，文件名为tb_name.ibd，表大小受操作系统文件大小的限制，一般2G
  • 通用表空间
  • 临时表空间
  • undo表空间

段：开启独占表空间后，会创建一个数据段(ibd文件)和索引段(frm文件)，段的空间大小随表的大小自动扩展，一个段至少会有一个区，段扩展的最小空间是区

  • 数据段：存储着索引树里的叶子结点的数据
  • 索引段：存储着索引树里的非叶子结点的数据
  • 回滚段

区：由64个连续的页组成，大小为1M = 64 * 16K(一个页的大小)，为保证区中页的连续性，扩展时会一次性申请4~5个区

页：大小为16k，一个数据页对应B+Tree的一个叶子节点；是MySQL InnoDB存储引擎内存和磁盘交互的最小单位，内存中有数据页和索引页

  • 支持ACID的事务，支持事物的四种隔离原则
  • 支持事务、行级锁及外键约束，支持写并发

需要对 insert、delete 、update、for update 操作进行加锁
添加行级锁是针对索引进行加锁

  • 按主键大小有序插入，不存储总行数
  • 主键采用聚集索引，辅索引的数据域存储主键的值，此时主键尽量采用自增主键；建议使用自增主键，避免插入时大量调整 B+树 结构，影响效率

1.3.3 MEMORY

所有数据都存储在内存中，数据处理速度极快，安全性不高，容易丢失

Notes:

  1. Implemented in the server, rather than in the storage engine. 在服务器中实现，而不是在存储引擎中实现

  2. Compressed MyISAM tables are supported only when using the compressed row format. Tables using the compressed row format with MyISAM are read only. 只有在使用压缩行格式时才支持压缩的MyISAM表。使用MyISAM压缩行格式的表是只读的。

  3. Implemented in the server via encryption functions. 在服务器端通过加密功能实现。

  4. Implemented in the server via encryption functions; In MySQL 5.7 and later, data-at-rest encryption is supported. 在服务器端通过加密功能实现；MySQL 5.7及以后版本支持数据静止加密。

  5. Support for foreign keys is available in MySQL Cluster NDB 7.3 and later. MySQL Cluster NDB 7.3及更高版本支持外键。

  6. Support for FULLTEXT indexes is available in MySQL 5.6 and later. 在MySQL 5.6及更高版本中提供了对全文索引的支持。

  7. Support for geospatial indexing is available in MySQL 5.7 and later. MySQL 5.7及更高版本支持地理空间索引。

  8. InnoDB utilizes hash indexes internally for its Adaptive Hash Index feature. InnoDB在内部利用哈希索引实现自适应哈希索引特性。

区别：

  • InnoDB 很多时候是行级锁，而 MyISAM 是表级锁，InnoDB 的并发高

  • InnoDB 支持事务，外键，数据完整性约束要强；MyISAM 不支持

  • 数据和索引的保存的文件不同：MyISAM 是分开保存，而 InnoDB 是保存到表空间

  • MyISAM 支持索引压缩，而 InnoDB 索引和数据是绑定保存不压缩，体积大

  • InnoDB 不支持 FULLTEXT 类型的索引(新版本的InnoDB 也支持)

  • InnoDB 中不保存表的具体行数，即执行select count(*) from table时，InnoDB要扫描一遍整个表来计算有多少行，但是MyISAM只要简单的读出保存好的行数即可。注意的是，当count(*)语句包含 where条件时，两种表的操作是一样的

  • 对于AUTO_INCREMENT类型的字段，InnoDB 中必须包含只有该字段的索引，但是在MyISAM 表中，可以和其他字段一起建立联合索引

  • 一个MyISAM表有三个文件：表结构文件、索引文件和数据文件；一个InnoDB存储引擎在一个文件空间（共享表空间，表大小不受操作系统限制，一个表可能分布在多个文件里），或多个文件空间中（设置为独立表空间，表大小受操作系统限制，一般为2G），受操作系统文件大小限制
