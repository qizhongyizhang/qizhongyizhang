写在前面：本文主要记录 Redis进阶-集群 的相关基础知识

1. 简介

Redis集群：实现对 Redis的水平扩容，即启动N个redis节点，将整个数据库分布存储在这N个节点中，每个节点存储总数据的1/N；通过分区（partition）来提供一定程度的可用性（availability）： 即使集群中有一部分节点失效或者无法进行通讯， 集群也可以继续处理命令请求

代理主机：通过代理服务器进行集群访问

无中心化集群：所有服务器都可以作为集群入口，服务器之间进行请求联通

cluster-enabled yes                  # 打开集群模式
cluster-config-file nodes-6379.conf  # 设定节点配置文件名
cluster-node-timeout 15000           # 设定节点失联时间，超过该时间（毫秒），集群自动进行主从切换

 --cluster-replicas 1 表示为集群中的每个主节点创建一个从节点：分配原则尽量保证每个主数据库运行在不同的IP地址，每个从库和主库不在一个IP地址上
 
slots：一个 Redis 集群包含 16384 个插槽（hash slot）， 数据库中的每个键都属于这 16384 个插槽的其中一个；集群使用公式 CRC16(key) % 16384 来计算键 key 属于哪个槽， 其中 CRC16(key) 语句用于计算键 key 的 CRC16 校验和

redis-cli -c -p port：
-c：集群方式连接

优点：实现扩容，分摊压力，无中心配置相对简单
缺点：不支持多键操作；不支持多键的Redis事务；不支持 lua脚本

集群配置：

  1. 端口号
  2. pid 文件名称
  3. log 文件名称
  4. dump.rdb 文件名称

2. 主从复制

2.1 介绍

主从复制：主机数据更新后根据配置和策略， 自动同步到备机的 master/slaver 机制，Master以写为主，Slave以读为主

优点：读写分离，性能扩展；容灾快速恢复

原理：

  1. Slave启动成功连接到 master后，向master发送一个 sync命令：请求进行数据同步
  2. Master接到 sync命令后，启动后台的存盘持久化进程，收集所有用于修改数据集命令， 在后台进程执行完毕后得到 rdb文件，master将整个 rdb数据文件传送到 slave
  3. 全量复制：slave服务器在接收到 master服务器的 rdb数据文件后，将其存盘并加载到内存中
  4. 增量复制：Master会持续将新的修改命令依次传给slave，完成同步；但是重新连接master后，会自动执行一次性完全同步（全量复制）

2.2 配置

  1. 配置开启：daemonize yes

  2. 配置主服务器：

include /myredis/redis_master.conf   # 指定配置文件路径
pidfile /var/run/redis_master.pid    # 配置数据文件位置
dbfilename dump_master.rdb           # 配置 rdb文件位置和名称
  1. 配置从服务器：
include /myredis/redis_slave.conf   # 指定配置文件路径
pidfile /var/run/redis_slave.pid    # 配置数据文件位置
dbfilename dump_slave.rdb           # 配置 rdb文件位置和名称

# replication
replicaof master_ip master_port     # 配置主机的ip和端口号，配置文件配置，服务器重启后，主从配置任然有效
masterauth master_pwd               # 配置主机的认证密码
  1. 启动服务：
redis-server redis_master.conf  # master

redis-server redis_slave.conf   # slave
slaveof master_ip master_port   # slave运行，命令行配置，服务器重启后，主从配置失效
  1. 查看服务器运行状况：info replication
    • role：服务器角色，分为 master|slave
    • connected_slaves：从机连接数

2.3 运行介绍

普通配置：

  • 主机 shutdown 后：从机默认不会升级成主机；重启后，仍然是之前配置的主机
  • 从机 shutdown 后：命令行配置重启后，默认会变成另外的主机，不会恢复成原来的从机（在配置文件中配置重启后依然是从机）；再次进行从机配置后，会完全加载主机的数据
  • 从机无法进行数据写操作

从机传递：从服务器指定特定从服务器为次主服务器；主服务器的操作会同步到特定从服务器中，其余从服务器同步特定从服务器的数据；当特定从服务器 shutdown 后，主服务器无法直接向其余从服务器同步操作

从机升主：主机宕机后，可通过slaveof no one 将当前从机升级为集群的主服务器，即其他从机会成为当前从机的从机；需要手动执行；原来的主机恢复，则会默认成为单独的新主机，原来的从机配置不会生效

哨兵模式：自动实现从机升主，主机重启后变为从机

  1. 主观下线：单个哨兵监测到主机宕机，系统认为单个哨兵主观认为主机宕机，不会马上进行 failover 故障转移过程
  2. 客观下线：当达到阈值数量的哨兵监测到主机宕机后，系统才会进行 failover 故障转移操作，按照一定算法挑选出新的主机，称为客观下线
# 新建哨兵配置文件
sentinel monitor mymaster master_ip master_port num
- mymaster：为监控对象起的服务器名称
- num：设置哨兵同意迁移的数量阈值

redis-sentinel  /myredis/sentinel.conf   # 指定哨兵配置文件路径

选举顺序：

  1. replica-priority num；选择 num 的最小值；num 的值越小，优先级越高，默认为 100
  2. 偏移量：选择偏移量最大；偏移量指获得原主机数据数量
  3. runid：选择 runid最小值；redis实例启动后都会随机生成一个40位的 runid

3. 分布式锁

分布式锁：一种跨数据库服务器的互斥机制，用来来控制共享资源的访问；分布式系统多线程、多进程并且分布在不同机器上，使原单机部署情况下的并发控制锁策略失效

实施方案：

  1. 基于数据库实现分布式锁
  2. 基于缓存（Redis等）：性能最高
  3. 基于Zookeeper：可靠性最高
